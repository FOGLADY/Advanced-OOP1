Module Units 1–12

This page presents the practical activities, artefacts and reflective learning developed throughout the twelve units of the Advanced Object-Oriented Programming module. Each unit includes an overview of the topic, a description of the artefact produced, the main learning achieved and a critical reflection on design choices, limitations and trade-offs. Supporting screenshots and technical evidence are provided in the accompanying Word document.

Unit 1: Introduction and Recap of Object-Oriented Programming

Unit Overview

This unit revisited the four core principles of object-oriented programming: abstraction, encapsulation, inheritance and polymorphism. It also reviewed classes, objects, constructors, properties, access control and the role of interfaces in creating maintainable software.

Artefact

I created a small Python AI model hierarchy using an abstract AIModel base class and two specialised subclasses: TextClassifier and ImageClassifier. Private attributes were used to protect model name, version and activation state. Properties provided controlled access, while each subclass implemented its own version of the shared predict() operation.

What I Learned

The activity demonstrated how a shared abstraction can provide a stable interface while allowing subclasses to vary their behaviour. Encapsulation reduced uncontrolled access to internal state, while polymorphism allowed client code to work with the base type rather than depend on a specific classifier.

Critical Reflection

Inheritance is useful when subclasses represent a genuine “is-a” relationship, but it can create tight coupling when used only for code reuse. A deep hierarchy would make the model family harder to modify, so composition would be preferable if behaviour needed to be assembled dynamically. This activity therefore showed both the value and the limits of inheritance-based design.

Unit 2: SOLID Principles of Object-Oriented Design

Unit Overview

This unit introduced the five SOLID principles: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation and Dependency Inversion. These principles guide the design of flexible, testable and maintainable object-oriented systems.

Artefact

I first created an over-responsible AIApplication class that performed model selection, prediction, result storage and console output. The program was then refactored into separate Predictor, ResultRepository and Reporter abstractions coordinated by a focused PredictionService. Text, image and audio predictor implementations could be substituted through the same interface.

What I Learned

The comparison showed that separating responsibilities makes code easier to extend, test and maintain. A new predictor or repository implementation could be introduced without modifying the existing service, demonstrating the Open/Closed and Dependency Inversion principles.

Critical Reflection

SOLID should be treated as a set of design guidelines rather than a requirement to create an interface for every class. Over-applying the principles to a small, stable script can result in unnecessary abstraction and a larger number of files. In this case, the abstractions were justified because model type, storage and reporting were realistic points of change.

Unit 3: Creational Design Patterns

Unit Overview

This unit examined creational design patterns including Singleton, Factory Method, Builder, Prototype and Abstract Factory. The main purpose of these patterns is to separate object creation from object use and reduce direct dependence on concrete classes.

Artefact

I implemented a Factory Method example in which specialised creator classes produced text and image AI services. I then implemented an Abstract Factory structure with local and cloud provider factories. Each concrete factory created a compatible family consisting of an AI model and its associated strategy or service configuration.

What I Learned

Factory Method centralised the decision about which concrete service should be created, while Abstract Factory ensured that related model families remained compatible. Client code could use the returned abstractions without knowing the exact implementation or deployment environment.

Critical Reflection

Factories are valuable when construction is complex, multiple product families are supported or environments may change. However, they introduce additional classes and indirect object creation. If only one implementation exists and creation is simple, direct construction may be clearer. The pattern should therefore be introduced in response to genuine variation rather than used automatically.

Unit 4: Structural Design Patterns

Unit Overview

This unit explored structural patterns and how they organise relationships between objects. The practical focus was the Decorator pattern, which adds responsibilities to an object without changing its original class.

Artefact

I created a base prediction service and wrapped it with SecurityDecorator, LoggingDecorator and TimingDecorator. Each decorator implemented the same component interface and delegated the core operation to the wrapped object. Security rejected invalid tokens, logging recorded request activity and timing measured execution duration.

What I Learned

The Decorator pattern allowed multiple behaviours to be combined at runtime without creating a separate subclass for every possible combination. The core service remained focused on prediction, while cross-cutting concerns were added independently.

Critical Reflection

Decorator supports the Open/Closed Principle, but a long chain of wrappers can make execution flow harder to follow. The order of decorators can also change behaviour; for example, logging outside the security decorator may record rejected requests, whereas logging inside it may not. The composition order must therefore be explicit and tested.

Unit 5: Behavioural Design Patterns

Unit Overview

This unit examined behavioural patterns that organise communication and responsibility between objects. The Strategy pattern was selected for the practical exercise because it encapsulates interchangeable algorithms behind a common interface.

Artefact

I created a PredictionStrategy abstraction with SentimentStrategy, KeywordStrategy and RiskDetectionStrategy implementations. A PredictionContext received a strategy through its constructor and could replace it at runtime.

What I Learned

The Strategy pattern removed algorithm-specific conditional logic from the context. Each algorithm could be developed and tested independently, while client code selected the most appropriate behaviour without modifying the context class.

Critical Reflection

Strategy is suitable when several meaningful algorithms solve the same problem or when behaviour must change dynamically. The trade-off is an increased number of small classes and a need to manage strategy selection. For a single simple algorithm that is unlikely to change, the pattern could become unnecessary over-engineering.

Unit 6: Concurrency and Thread-Safe Programming

Unit Overview

This unit examined multithreading, race conditions, shared mutable state, synchronisation and deadlock. The main objective was to ensure that concurrent operations remained correct when several threads accessed the same objects.

Artefact

I developed a thread-safe banking system containing BankAccount and TransactionSimulator classes. Deposits, withdrawals, balance reads and transfers were protected with RLock. Transfers acquired two account locks in a fixed order based on account number to reduce deadlock risk.

What I Learned

The activity demonstrated that apparently simple balance updates are not atomic and can produce inconsistent results when threads interleave. Locking protected account invariants, while ordered lock acquisition prevented circular waiting during transfers.

Critical Reflection

Synchronisation protects correctness but can reduce concurrency if locks are held for too long. Critical sections should therefore be kept small and the locking policy documented. The prototype used floating-point values for simplicity, whereas a production financial system should use decimal arithmetic, persistent transaction records, authentication, audit trails and recovery mechanisms.

Unit 7: Secure Coding

Unit Overview

This unit considered how insecure implementation decisions can expose otherwise functional object-oriented systems. Topics included validation, injection attacks, password storage, secure comparison, error handling and defence in depth.

Artefact

I created an intentionally vulnerable SQLite login component that constructed SQL statements through string concatenation and stored a password in plain text. A crafted input successfully bypassed the login check. I then refactored the component to use parameterised queries, username and password validation, PBKDF2 password hashing with a unique salt, hmac.compare_digest() and more specific exception handling.

What I Learned

The comparison showed that security must be enforced at input, domain and data-access boundaries. Parameterised queries removed the injection path, while salted password hashing reduced the consequences of a database disclosure.

Critical Reflection

The refactored example is more secure but does not represent a complete authentication platform. A production solution would also require session management, rate limiting, secrets management, multi-factor authentication, access logging and careful separation between internal errors and user-facing messages. Secure coding is therefore an ongoing architectural concern rather than one isolated fix.

Unit 8: Refactoring and Code Smells

Unit Overview

This unit addressed common code smells and disciplined refactoring. Refactoring improves the internal structure of code while preserving its observable behaviour.

Artefact

I began with an OrderProcessor containing a long process_order() method responsible for validation, discount selection, tax calculation and receipt formatting. The code also relied on primitive strings and magic numbers. I refactored it into immutable Order and OrderTotals data classes, a CustomerType enumeration and separate OrderValidator, DiscountPolicy, PriceCalculator, ReceiptFormatter and OrderService classes.

What I Learned

The refactoring separated responsibilities, removed magic values from the main workflow and made the business rules easier to test. The identical receipt before and after the change confirmed that the external behaviour had been preserved.

Critical Reflection

The refactored design contains more classes, so it would be excessive for a disposable script. In a maintainable business system, however, the added structure supports focused changes and independent testing. Refactoring should therefore improve a real maintenance problem rather than simply increase abstraction.

Unit 9: Object-Oriented Software Architecture

Unit Overview

This unit examined monolithic, layered and microservice architectures and considered how architectural choices influence scalability, maintainability, performance and security.

Artefact

I implemented a small layered e-commerce architecture. The domain layer contained Product and Order objects. The data access layer provided product and order repository abstractions and in-memory implementations. The service layer enforced business rules, while an OrderController represented the presentation layer.

What I Learned

The layered structure created clear dependency boundaries and allowed the storage mechanism to be replaced without changing the business service. The controller delegated business decisions rather than implementing them directly.

Critical Reflection

A layered architecture was appropriate for this application because it offered separation without the operational complexity of distributed services. Microservices can scale teams and workloads independently, but they introduce network failure, distributed data management, deployment overhead and a larger security boundary. Architecture should therefore respond to actual scale and change requirements rather than follow trends.

Unit 10: Test-Driven Development and Behaviour-Driven Development

Unit Overview

This unit introduced the TDD cycle of Red, Green and Refactor, the importance of unit testing, behaviour-oriented test design and mocking techniques for isolating dependencies.

Artefact

I developed an AI content moderation service around an AIModelClient abstraction. Tests defined expected behaviour for safe content, high-confidence malware content, low-confidence results, empty input, invalid confidence values and AI provider failure. unittest.mock.Mock replaced the AI dependency, allowing deterministic tests without a live model or external API.

What I Learned

Writing tests around observable outcomes encouraged a small, explicit public interface and made edge cases part of the design rather than an afterthought. Mocking made the tests fast, repeatable and independent from network availability.

Critical Reflection

Mocks verify the contract represented in a test but may not behave exactly like a real AI provider. Unit testing must therefore be complemented by integration testing at system boundaries. The practical implementation used descriptive unittest methods rather than a dedicated BDD framework, but the tests still expressed behaviour in terms of expected outcomes rather than implementation details. Ethical concerns such as bias, explainability and confidence calibration would also need dedicated tests in a production AI system.

Unit 11: Dependency Injection and Inversion of Control

Unit Overview

This unit focused on Dependency Injection and Inversion of Control as methods of separating object behaviour from dependency creation. It also considered the benefits and drawbacks of container-based frameworks such as Spring and Google Guice.

Artefact

The Unit 10 moderation artefact also provided a direct Dependency Injection example. ModerationService received an AIModelClient through constructor injection rather than constructing a concrete client internally. The demonstration supplied a local fake implementation, while the test suite injected a mock implementing the same contract.

What I Learned

Constructor injection made dependencies visible, reduced coupling and prevented hidden creation logic inside the service. Control over object construction moved to the application composition code, demonstrating Inversion of Control.

Critical Reflection

A full DI framework can automate large object graphs, scopes and lifecycle management. In a small Python application, explicit constructor injection is often clearer and easier to debug. Introducing a container too early can hide dependencies behind configuration and make runtime behaviour more difficult to trace. The complexity of the dependency-management mechanism should match the size of the application.

Unit 12: Recent Developments and Review

Unit Overview

The final unit consolidated advanced OOP, SOLID principles, design patterns, secure coding, architecture, concurrency and testing through a Capstone Project addressing an AI and cybersecurity-related problem.

Capstone Artefact

I designed and implemented the Secure AI Content Moderation Platform. The system accepts textual content requests and returns ALLOW, REVIEW or BLOCK decisions. The platform is divided into separate domain, service, repository, model-provider and presentation responsibilities.

The Strategy pattern provides interchangeable moderation algorithms. Decorator adds token validation, logging and execution timing. Abstract Factory creates compatible local or cloud model families. Visitor generates an audit summary without changing the moderation result classes. Constructor injection connects the model client, strategy, repository and Model Registry.

The application also uses ThreadPoolExecutor for concurrent request processing, while the Model Registry and result repository protect shared state with RLock. Domain objects validate requests and prediction confidence values. The AI dependency is mocked during testing, so tests run without a live model or external API.

What I Learned

The Capstone showed how patterns and principles become more useful when they solve related design problems within one system. Rather than treating Strategy, Decorator, Abstract Factory and Visitor as isolated examples, I used them to support replaceable decisions, cross-cutting security, provider selection and audit reporting. The thread-safe Model Registry also introduced the idea of maintaining model identity, version and provider metadata.

Critical Reflection

The modular design improves extensibility and testing, but it also contains more abstraction than a basic rule-based script would require. The current model clients are deterministic local implementations rather than genuine machine-learning models. A production version would need authenticated API communication, encrypted persistence, rate limiting, monitoring, bias evaluation, explainability, model drift detection and human review workflows. A Feature Store could be introduced if reusable, versioned input features were required across training and inference, while the Model Registry would need persistent storage and deployment-stage metadata. These additions would increase operational complexity and should only be introduced when justified by real scale.
